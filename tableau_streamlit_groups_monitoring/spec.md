## Problem Statement

Leadership and compliance need visibility into Tableau Server group memberships. When asked "who's in what group?", nobody can produce the answer quickly. Today the response is "we'll get back to you" or guesswork. No scripts, no exports, no process exists to answer this question. This creates compliance risk and makes the admin team look unprepared.

## Demand Evidence

- Audit/compliance pressure from manager/director level
- The question "who has access to what?" comes up regularly and cannot be answered
- The current answer is "we'll get back to you" which is a governance red flag
- No existing internal tooling addresses this gap

## Status Quo

Nothing. When compliance asks about Tableau group memberships, the organization has no process. No scripts, no exports, no dashboard. The Tableau Server admin UI allows manual click-through of groups one-by-one, but this is too slow and non-exportable for compliance requests.

## Target User & Narrowest Wedge

**Primary user:** The Tableau Server admin who fields compliance and management questions about group memberships. They need to answer "who's in group X?" and "what changed since last month?" in under 30 seconds with a CSV to attach.

**Secondary consumer:** Manager/director who receives the exported answers (CSV, screenshot) but does not interact with the tool directly.

**Narrowest wedge:** A Streamlit app that shows current group memberships with search/filter and CSV export, plus a diff view showing membership changes between snapshot dates.

## Constraints

- Tableau Server (on-prem/self-hosted), not Tableau Cloud
- REST API access via Personal Access Token (PAT) or admin credentials
- Python/Streamlit stack (org can run a Python web app internally)
- No real-time event streaming needed on day one — point-in-time snapshots are sufficient
- SQLite for storage — zero external infrastructure requirements
- Must answer compliance questions fast, with exportable evidence

## Premises

1. **Primary user is the admin** — the person building this tool is the one who will use it to answer compliance/management questions. The manager consumes exports, not the dashboard. (Revised based on cross-model challenge.)
2. **Group membership is the core data** — who's in which group, when membership changed, unexpected additions/removals. Not per-workbook permissions.
3. **Tableau Server (on-prem)** — REST API access via PAT.
4. **Streamlit is the delivery mechanism** — the org can host a Python web app internally.
5. **Point-in-time snapshots are sufficient** — no real-time streaming needed. Periodic snapshots with change detection is the wedge.

## Cross-Model Perspective

An independent Claude subagent reviewed this design cold. Key findings:

- **Steelman:** "A compliance-critical visibility gap disguised as a 'nice to have' dashboard. Done right, this becomes the system of record for Tableau access governance, which is a gap no commercial tool fills."
- **Key insight:** The real product is an "answer engine for compliance questions," not a passive monitoring dashboard. Optimize for search and export, not glanceability.

## Approach

**Approach A: Snapshot + Diff Engine.** It ships in a day, answers the compliance question immediately, and the other approaches are natural extensions to layer on once value is proven.

### Architecture

```
┌─────────────────┐     ┌──────────────────────┐     ┌──────────────────┐
│  Tableau Server  │────>│ snapshot.py          │────>│  SQLite DB       │
│  (REST API/TSC)  │     │  1. Take snapshot    │     │  - snapshots     │
└─────────────────┘     │  2. Compute diff     │     │  - group_members │
                         │     (vs. previous)   │     │  - changes       │
                         └──────────────────────┘     └────────┬─────────┘
                                                               │
                                                      ┌────────▼─────────┐
                                                      │  Streamlit App   │
                                                      │  - Search/Filter │
                                                      │  - Diff View     │
                                                      │  - CSV Export    │
                                                      └──────────────────┘
```

### Execution Flow

1. **Admin** runs `python snapshot.py` manually when a fresh snapshot is needed
2. `snapshot.py` authenticates via PAT, pulls all groups and members using `TSC.Pager()` for full pagination, writes to `group_members` table within a SQLite transaction
3. On success, `snapshot.py` imports `diff.py` and calls its diff function to compute changes against the previous snapshot, writing to `membership_changes`. (`diff.py` is also CLI-invocable for ad-hoc comparisons between arbitrary snapshot dates.) If the API call fails mid-run, the transaction rolls back (no partial snapshots).
4. The **Streamlit app** reads from SQLite on demand. It does not run snapshots or diffs.

**Error handling:** If the Tableau Server is unreachable or the PAT has expired, `snapshot.py` logs the error and marks the snapshot status as "failed". No partial data is written (transaction rollback). PAT expiration is detected by a 401 response from TSC, which should prompt the admin to regenerate the token.

### Diff Algorithm

The diff compares two snapshots by computing set differences on `(group_id, user_id)` tuples:

- **Added:** present in current snapshot but not in previous → `change_type = "added"`
- **Removed:** present in previous snapshot but not in current → `change_type = "removed"`

Note: `site_role` changes within a group (e.g., user promoted from Viewer to Explorer) are **not** tracked as membership changes in v1. This could be added as a "role_changed" change type in a future iteration if compliance requires it.

### Data Model

**`snapshots` table:**
- `id` (INTEGER PRIMARY KEY)
- `timestamp` (DATETIME) — when the snapshot was taken
- `status` (TEXT) — success/failure

**`group_members` table:**
- `id` (INTEGER PRIMARY KEY)
- `snapshot_id` (INTEGER FK → snapshots.id)
- `group_name` (TEXT)
- `group_id` (TEXT) — Tableau's internal group ID
- `user_name` (TEXT)
- `user_id` (TEXT) — Tableau's internal user ID
- `site_role` (TEXT) — user's site role at time of snapshot
- `domain_name` (TEXT) — AD/LDAP domain if applicable

**`membership_changes` table:**
- `id` (INTEGER PRIMARY KEY)
- `detected_at` (DATETIME)
- `group_name` (TEXT)
- `group_id` (TEXT) — Tableau's internal group ID (rename-resilient)
- `user_name` (TEXT)
- `user_id` (TEXT) — Tableau's internal user ID (rename-resilient)
- `change_type` (TEXT) — "added" or "removed"
- `previous_snapshot_id` (INTEGER FK)
- `current_snapshot_id` (INTEGER FK)

**Indexes:**
- `group_members`: `(snapshot_id, group_name)`, `(snapshot_id, user_name)`
- `membership_changes`: `(current_snapshot_id)`, `(group_name)`

**Data growth:** At 50 groups x 20 users = 1,000 rows per snapshot. Daily snapshots = ~365K rows/year. SQLite handles this comfortably. The Streamlit dataframe display should use pagination if total rows exceed a few thousand.

### Streamlit Pages

**Page 1: Current State**
- Dropdown to select snapshot date (defaults to latest)
- Search bar filtering across group name and user name
- Filterable table: Group Name | User Name | Site Role | Domain
- "Export CSV" button for the current filtered view
- Summary stats: total groups, total users, total memberships

**Page 2: Changes**
- Date range picker (from snapshot → to snapshot)
- Table: Group Name | User Name | Change Type (Added/Removed) | Detected At
- Color-coded: green for additions, red for removals
- "Export CSV" button for the change report
- Summary: X additions, Y removals across Z groups

## Distribution Plan

Internal tool. Runs on the admin's machine or a shared internal VM. Streamlit serves on localhost or internal network. No external distribution needed.

Deployment: `pip install -r requirements.txt && streamlit run app.py`. Run `python snapshot.py` manually to capture data.

**Access control:** If deployed on a shared VM (not localhost), restrict access at the network level (firewall rules, VPN requirement) since the app exposes group membership data which is itself sensitive. Streamlit Community Cloud auth or a reverse proxy with basic auth are options if network-level restriction isn't sufficient.

## The Assignment

The goal is to write an MVP streamlit app to handoff.

```python
import tableauserverclient as TSC

server = TSC.Server('https://your-tableau-server.com', use_server_version=True)
auth = TSC.PersonalAccessTokenAuth('your-pat-name', 'your-pat-secret', site_id='')
with server.auth.sign_in(auth):
    # Use TSC.Pager to handle pagination (server.groups.get() only returns the first page)
    all_groups = list(TSC.Pager(server.groups))
    print(f"Total groups: {len(all_groups)}")
    for g in all_groups:
        print(f"Group: {g.name} (ID: {g.id})")

    # Test member listing for one group
    # NOTE: populate_users() fetches the first page only by default.
    # For the full implementation, use RequestOptions with pagination
    # to handle groups with 100+ members. This sample is for quick validation.
    if all_groups:
        server.groups.populate_users(all_groups[0])
        print(f"\nMembers of '{all_groups[0].name}' (first page):")
        for user in all_groups[0].users:
            print(f"  User: {user.name} (Role: {user.site_role})")
```

**Important:** `server.groups.get()` only returns the first page of results. Always use `TSC.Pager()` to iterate all groups. `populate_users()` also paginates, use `RequestOptions` with incrementing page numbers in the full implementation to ensure all members of large groups are captured.

If that prints your groups and at least one group's members, you're ready to build.
